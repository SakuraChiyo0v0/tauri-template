## Context

运行时依赖解析器已经产生稳定的拓扑激活顺序：依赖提供者先激活，消费者后激活；计划变化会重载 WebView，模块停用会调用 `deactivate` 并释放 Host SDK。当前缺口是 SDK 没有受控的跨模块调用边界，因此依赖只能决定加载顺序，不能让模块协作。

三个真实模块必须继续与基座源码分离。它们通过独立仓库、版本化 SDK 类型和 `.mtp` 包接入，不新增基座业务页面，也不把安装实例或打包产物提交到基座。

## Goals / Non-Goals

**Goals:**

- 让模块显式声明并提供小型服务，让已声明依赖的消费者进行异步调用。
- 保持参数、返回值、权限和生命周期边界清楚，避免模块取得底座内部注册表或另一个模块的 SDK。
- 用便签、统计面板和快捷启动器验证数据库、依赖、服务调用、双语、托盘、快捷键与 URL 打开能力。
- 让模块模板成为 SDK V4 的可复制起点。

**Non-Goals:**

- 不提供跨 WebView、跨进程、网络或远程模块服务。
- 不提供任意 JavaScript 对象、DOM、函数引用或流式数据共享。
- 不提供服务事件订阅、后台守护进程、在线模块市场或自动更新源。
- 不把三个验证模块编译进基座或设为内置模块。

## Decisions

### Host SDK V4 是 V3 的严格超集

新增 `sdkVersion: 4`，保留 V2/V3 运行语义。V4 继承 V3 的数据库和受控原生能力，再增加 `services.expose`、`services.call` 与 `services.available`。采用新版本而不是静默扩展 V3，可以让 manifest、模板和运行时对公共 ABI 的能力边界保持一致。

### 服务必须由提供者清单声明

schema V2 清单新增 `services.provides`，其中是当前模块允许注册的服务 ID。`expose` 只能注册清单列出的服务，重复注册失败。服务方法仍由模块代码提供，不把方法清单写进 manifest，避免每次内部方法调整都迫使底座理解业务协议。

### 消费者授权复用模块依赖声明

`call(providerId, serviceId, method, input)` 只允许访问调用模块清单中 `required` 或当前已选择的 `optional` 依赖。服务总线不再维护第二套授权关系；依赖解析负责版本与激活顺序，服务总线负责运行时调用检查。模块不能调用任意已安装模块，也不能调用底座内部服务。

### 服务边界只接受受限 JSON 值

输入和输出仅允许 `null`、布尔值、有限数字、字符串、数组和普通字符串键对象。调用前后均进行深度复制和大小检查，拒绝函数、类实例、循环引用、危险原型键和超限数据。这样消费者不会获得提供者的可变对象或执行引用。

### 服务注册跟随 Host SDK 生命周期

服务注册表只存在于当前 WebView。提供者激活失败、停用或计划重载时，底座释放其 SDK 并删除全部服务；调用已释放或未注册的服务会返回明确错误。依赖拓扑保证正常启动时提供者先完成注册，消费者随后才能调用。

### 三个模块按单一职责验证能力

- `local-notes` 使用隔离 SQLite 提供便签 CRUD 页面，并暴露 `notes.v1` 服务。
- `notes-dashboard` 必需依赖 `local-notes`，通过服务读取统计和最近便签，不直接访问提供者数据库。
- `quick-launcher` 使用 SQLite 保存 HTTPS 快捷入口，并通过托盘与全局快捷键打开当前入口。

三个仓库从当前模块模板派生，移除模板远程地址，各自保留独立 Git 历史。后续只有用户创建对应 GitHub 仓库后才配置远程和推送。

## Risks / Trade-offs

- [提供者升级改变业务方法] → 服务 ID 带主版本后缀，例如 `notes.v1`；破坏性协议使用新服务 ID，并通过模块依赖版本协调升级。
- [同一 WebView 不能形成强安全隔离] → 服务总线只减少误用和耦合，不声称隔离恶意 JavaScript；不暴露 Tauri invoke、内部 store 或原始对象。
- [调用大对象阻塞 UI] → 限制嵌套深度和序列化大小；大数据由提供者分页返回。
- [消费者激活时服务尚未注册] → 保持依赖拓扑顺序，并把未注册服务视为消费者激活错误进入现有恢复流程。
- [真实模块扩大迭代范围] → 每个模块只实现一个页面和最短关键路径，不加入搜索、同步、导入导出或复杂编辑器。

## Migration Plan

1. 基座和 Rust manifest 校验增加 SDK V4 与服务声明。
2. 模板升级到 SDK V4，并先通过类型、服务总线和打包关键测试。
3. 创建三个独立模块并生成 `.mtp`。
4. 用真实包验证提供者先安装、消费者等待依赖、依赖补齐后激活，以及快捷启动器权限批准路径。
5. 若服务总线出现不可接受问题，可卸载验证模块并回退基座提交；模块数据库按现有策略保留，用户可显式清理。

## Open Questions

- 服务事件订阅和跨模块推送暂不进入 V4；真实模块验证后再决定是否需要 V5 事件通道。
