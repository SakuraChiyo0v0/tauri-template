## Why

当前运行时模块彼此独立安装和激活，无法表达“模块 B 需要模块 A 的某个版本”，也无法阻止升级、回滚、停用或卸载破坏其他模块。现在底座仍处于早期，应先建立依赖图和全局激活边界，再开发真正相互协作的业务模块。

## What Changes

- 运行时模块清单可声明必需依赖和可选依赖，并为每个依赖指定语义版本范围。
- 安装包在依赖尚未满足时仍可安全落盘，但进入“等待依赖”状态，不会成为可激活模块。
- Rust 维护全局、原子保存的运行时模块激活计划，统一记录启用状态、选择版本、依赖阻塞和计划代次。
- 依赖解析器检测缺失依赖、版本不兼容、循环依赖和不可满足的版本组合，并产生可供 UI 展示的结构化诊断。
- 模块启用、升级、回滚和卸载在提交前验证整张依赖图；默认拒绝破坏已启用依赖者的操作，不执行静默级联。
- 前端根据 Rust 返回的已解析激活计划按拓扑顺序加载模块；基础模块失败时只阻止其依赖者，其他独立子图继续运行。
- 模块管理页面展示依赖、被依赖关系、等待/阻塞原因和受影响模块。
- 现有不声明依赖的 V1 `.mtp` 模块继续兼容；仍只允许每个模块激活一个版本。
- 不加入远程依赖下载、多版本并存、自动级联卸载或跨模块内部源码导入。

## Capabilities

### New Capabilities

- `runtime-module-dependency-resolution`: 定义依赖声明、版本匹配、图解析、拓扑顺序、阻塞传播和结构化诊断。

### Modified Capabilities

- `module-package-lifecycle`: 安装、启用、升级、回滚和卸载必须通过全局依赖计划验证，并支持已安装但等待依赖的版本。
- `runtime-module-contributions`: 启动发现改为消费已解析计划，按拓扑顺序激活，并隔离依赖失败。

## Impact

- TypeScript 与 Rust 的模块清单、安装状态和命令返回结构会扩展。
- Rust 模块存储从逐模块活动状态演进为“不可变版本目录 + 全局激活计划”，并需要兼容现有 `state.json` 数据。
- 前端启动加载器、运行时模块 API、模块注册表和模块管理页面需要读取依赖状态。
- 需要新增图解析、状态迁移、生命周期事务、拓扑激活和 UI 状态测试。
- Host SDK V1 的日志、设置和主题能力保持不变；跨模块服务调用不在本变更范围内。
