## Context

当前每个运行时模块在自己的 `state.json` 中保存活动版本、上一版本和版本哈希；前端从 Rust 获取模块列表后按名称顺序逐个激活，启用状态则保存在 WebView `localStorage`。这种模型无法回答以下问题：

- 模块所需的另一个模块是否存在且版本兼容；
- 多个模块应以什么顺序激活；
- 升级或回滚一个模块是否会破坏依赖者；
- 两个需要协调升级的模块如何在不产生中间坏状态的情况下切换；
- 基础模块失败后哪些模块应该停止、哪些独立模块仍可继续。

单纯在清单上添加 `dependencies` 并按当前逐模块状态操作，会在协调升级时形成死锁。例如 B1 依赖 A1，而 B2 依赖 A2：先切 A2 会破坏 B1，先切 B2 又不满足 A1。需要把“包已安装”和“版本已进入活动组合”分成两个阶段。

## Goals / Non-Goals

**Goals:**

- 保持现有无依赖 `.mtp` 包兼容。
- 支持必需和可选模块依赖及语义版本范围。
- 允许依赖未满足的包安全落盘并显示为等待状态。
- 使用一个原子全局计划表达所有期望启用模块及其当前选择版本。
- 在安装、启用、升级和回滚时求解一组兼容版本，并阻止破坏依赖图的停用和卸载。
- 按稳定拓扑顺序激活；一个子图失败不影响无关模块。
- 将缺失、版本冲突、循环和上游失败变成结构化、可展示的诊断。
- 迁移现有逐模块状态，并在升级后首个启动周期吸收前端保存的运行时启用状态。

**Non-Goals:**

- 不提供远程依赖下载或自动安装依赖。
- 不支持同一模块多个版本同时激活。
- 不允许模块直接导入另一个模块的入口或内部文件。
- 不加入跨模块服务调用 API；真正出现调用需求时再版本化扩展 Host SDK。
- 不执行静默级联停用或卸载。
- 不实现通用 SAT 包管理器；求解范围仅限本地已安装版本和单个活动版本约束。

## Decisions

### 1. 清单使用显式依赖数组并保持 schema/SDK V1 兼容

清单新增可选字段：

```json
{
  "dependencies": {
    "required": [
      { "id": "data-provider", "version": "^1.2.0" }
    ],
    "optional": [
      { "id": "export-tools", "version": ">=1.0.0, <2.0.0" }
    ]
  }
}
```

依赖模块 ID 不得等于自身，必需与可选列表中不得重复。版本范围由 Rust `semver::VersionReq` 校验；TypeScript 执行结构和基本格式校验，Rust 是安装可信边界。

依赖字段缺省等同于空列表，因此现有 schemaVersion 1 包继续工作。Host SDK 没有增加能力，sdkVersion 保持 1。

备选方案是以能力名称而不是模块 ID 依赖。能力契约更松耦合，但当前没有跨模块服务 API，提前引入会产生不可验证的抽象；本阶段先解决真实模块包的存在、版本和生命周期关系。

### 2. 分离不可变版本目录、模块目录元数据和全局激活计划

目录演进为：

```text
modules/
├─ activation-plan.json
├─ <module-id>/
│  ├─ state.json
│  └─ versions/<version>/...
```

模块 `state.json` 继续保存已安装版本哈希和激活错误，用于兼容迁移；全局 `activation-plan.json` 成为活动组合与运行时启用状态的唯一权威：

```text
generation
desiredEnabled[moduleId]
selectedVersions[moduleId]
previousSelectedVersions[moduleId]
diagnostics[moduleId][]
```

安装新包先原子加入版本目录和模块版本目录元数据，再尝试生成新计划。计划不可解时，新版本保留为待选版本，旧计划不变。写入全局计划使用同目录临时文件、刷新和原子替换。

这样即使进程在包落盘后中断，最多留下尚未激活的完整版本，不会产生半张依赖图。

### 3. 使用确定性的有限回溯求解单活动版本组合

求解输入是：

- 所有期望启用模块；
- 每个模块已安装且未被激活阻止的版本及清单；
- 当前选择版本；
- 当前操作希望优先选择的版本。

候选顺序为：操作指定版本 → 当前选择版本 → 其余版本从高到低。求解器逐模块选择一个版本，检查必需依赖范围，并在有限候选集合上回溯。找到结果后再检测必需依赖环并生成稳定拓扑顺序；同层模块按 ID 排序，保证测试和启动结果可重复。

可选依赖缺失或版本不匹配不阻止模块；只有存在兼容且已选择的可选依赖时，才添加拓扑边。

不采用贪心“永远选择最高版本”，因为它无法处理协调升级，也可能无故切换未参与操作的模块。有限回溯的规模受本地模块数和每模块保留版本数约束，V1 包限制下可接受；后续可增加版本保留上限。

### 4. 计划可包含期望启用但未被选择的等待模块

首次安装模块缺少依赖时：

- 包与版本元数据保留；
- `desiredEnabled[moduleId] = true`；
- `selectedVersions` 不包含该模块；
- `diagnostics` 包含缺失或版本冲突；
- 前端不加载其入口。

后续安装依赖或兼容版本时重新求解，可能一次性把多个等待版本纳入新计划。安装结果需要返回“包已保存”和“计划是否改变”，UI 不把等待状态误报为安装失败。

### 5. 运行时启用状态移入 Rust，源码模块状态保持现状

只有运行时模块的启用状态进入全局计划；内置源码模块继续使用现有前端状态。

为保留已有安装的前端启用选择，启动发现命令接受一次 `legacyDisabledModuleIds`。当全局计划不存在时，Rust 从旧 `state.json` 活动版本建立初始计划并应用这些禁用项；计划创建后忽略该迁移输入。前端随后清理这些运行时模块的旧本地启用键，避免双重权威。

### 6. 生命周期命令返回计划结果和结构化影响

运行时命令围绕计划操作：

- 安装：保存版本，尝试求解；允许等待状态。
- 启用：设置期望启用并求解；无解时保留期望状态和诊断，但不激活。
- 停用：如果存在已启用依赖者则拒绝，并返回依赖者 ID。
- 回滚：强制目标模块优先选择其上一计划版本，只有整图可解才提交。
- 卸载：如果存在已启用或等待中的必需依赖者则拒绝；不静默级联。
- 激活失败：阻止失败版本，重新求解计划；有兼容组合则提交并要求刷新，否则传播上游失败诊断。

错误采用稳定代码、模块 ID、关联模块和版本字段，UI 文案由前端生成。普通参数/IO 错误仍可返回字符串错误。

### 7. 前端只消费已解析计划并按拓扑顺序激活

Rust 返回：

- 所有已安装模块及版本信息；
- `desiredEnabled`、选择版本、状态和诊断；
- 可激活模块的拓扑顺序；
- 每个模块的直接依赖和直接依赖者。

前端不重复做版本求解。它严格按拓扑顺序激活选择版本。若 provider 激活失败，本轮跳过所有可达依赖者并标记 `upstream_activation_failed`，但继续处理不相关分支。失败上报 Rust 后，若计划代次变化则刷新 WebView。

### 8. 计划级上一版本支持依赖安全回滚

每次选择版本组合改变时，旧 `selectedVersions` 整体保存为 `previousSelectedVersions`。用户对某模块回滚时，取上一计划中该模块的版本作为优先目标，但仍重新求解并验证所有当前期望启用模块。

如果回滚需要同时改变其他模块，命令先返回影响预览并拒绝隐式提交；本阶段仅允许不改变其他已选模块的安全回滚。协调回滚可作为后续显式批量计划功能。

## Risks / Trade-offs

- **[全局计划迁移错误可能改变现有活动版本]** → 初始计划优先沿用每个旧状态的活动版本；迁移测试覆盖禁用模块、损坏状态和重复启动幂等性。
- **[有限回溯在大量版本组合下变慢]** → 确定性候选顺序、提前约束剪枝，并设置最大搜索节点数；超限返回可诊断错误而不是卡住启动。
- **[包已安装但未激活可能让用户困惑]** → UI 明确区分“已安装/等待依赖/已启用/被阻止”，显示缺失模块和版本范围。
- **[前端与 Rust 同时判断依赖导致漂移]** → Rust 是唯一解析器；TypeScript 只验证清单形状和消费结构化计划。
- **[停用基础模块的用户操作受限]** → 默认安全拒绝并列出依赖者；后续可增加明确的级联预览事务，不在本次静默处理。
- **[旧模块状态仍存在重复字段]** → 迁移后全局计划为权威，旧 active 字段只用于迁移和兼容读取；后续格式版本可清理。

## Migration Plan

1. 扩展 Rust/TypeScript 清单并保持空依赖兼容。
2. 引入纯依赖解析器和计划类型，以内存测试覆盖协调升级、缺失、冲突、循环和确定性顺序。
3. 加入全局计划读写与旧状态迁移，不改变前端加载器。
4. 将安装、启用、停用、回滚、失败恢复和卸载命令改为计划事务。
5. 前端启动改为传入旧禁用状态、消费计划和拓扑顺序。
6. 更新模块管理 UI、示例清单和 AI 文档。
7. 使用临时旧版应用数据目录验证迁移，并使用真实多模块 `.mtp` 组合验证等待、协调升级和阻止卸载。

回退代码时保留 `activation-plan.json` 和版本目录；旧底座仍可读取各模块旧状态中最后一次同步的活动版本。实现期间每次计划提交同时更新兼容 active 字段，以提供这一回退路径。

## Open Questions

- 跨模块服务调用需要怎样的能力命名和失效语义，将由第一个确有调用需求的模块单独提案；本变更只保证依赖存在、版本和激活顺序。
- 显式级联停用、批量计划预览和协调回滚可复用本次影响分析结果，但不在本次加入修改操作。
