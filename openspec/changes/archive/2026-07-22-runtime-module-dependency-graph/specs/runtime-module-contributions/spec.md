## MODIFIED Requirements

### Requirement: 底座必须发现运行时模块
底座启动时 MUST 从 Rust 安装服务读取所有已安装运行时模块以及已解析的全局激活计划。底座 SHALL 只注册计划选择且未被依赖阻止的模块贡献，并使用计划状态展示等待或失败模块；运行时模块不得要求修改应用外壳或静态模块注册文件。

#### Scenario: 启动时发现完整依赖图
- **WHEN** 应用数据目录中存在状态完整且依赖可解的活动模块
- **THEN** 模块管理器显示所有已安装模块，并按计划拓扑顺序激活选择模块和聚合贡献

#### Scenario: 发现等待依赖模块
- **WHEN** 已安装模块期望启用但未被全局计划选择
- **THEN** 模块管理器显示该模块及结构化依赖诊断，侧边栏和设置不显示其贡献

#### Scenario: 无运行时模块时启动
- **WHEN** 应用数据目录中没有安装运行时模块
- **THEN** 内置系统模块和日志模块继续正常初始化，且底座不显示虚构的演示模块

#### Scenario: 迁移旧前端启用状态
- **WHEN** 首次创建全局计划且旧前端状态中存在已停用运行时模块
- **THEN** 启动引导在激活入口前把这些模块作为禁用迁移给 Rust，之后以 Rust 计划为唯一运行时启用状态

### Requirement: 启停必须同步改变贡献和生命周期
用户启用或停用运行时模块时，系统 MUST 通过 Rust 全局计划改变其期望启用状态并重新解析依赖图。停用不得破坏仍期望启用的必需依赖者；重新启用只有在进入选择计划并成功激活后才能恢复贡献。

#### Scenario: 停用无依赖者运行时模块
- **WHEN** 用户在模块管理器关闭一个不被其他期望启用模块必需依赖的模块
- **THEN** 新计划移除该模块选择，其导航和设置消失，并调用可选停用生命周期

#### Scenario: 停用必需 provider
- **WHEN** 用户关闭一个仍有期望启用依赖者的模块
- **THEN** 系统拒绝停用、显示依赖者列表，现有页面和设置保持不变

#### Scenario: 重新启用且依赖满足
- **WHEN** 用户重新启用模块且解析器找到完整兼容组合
- **THEN** 系统提交新计划、按拓扑顺序激活，并在成功后恢复其导航和设置贡献

#### Scenario: 重新启用但依赖缺失
- **WHEN** 用户重新启用模块但必需依赖仍不满足
- **THEN** 系统保存期望启用状态并显示等待依赖，不执行入口或恢复贡献

### Requirement: 版本切换必须清理旧运行上下文
安装、依赖解析、启停、回滚、激活失败恢复或卸载导致全局计划选择版本改变后，底座 SHALL 重新加载 WebView，并只按新计划代次和拓扑顺序导入入口，避免在同一 JavaScript 上下文混用版本或保留失效依赖引用。

#### Scenario: 协调升级后重新加载
- **WHEN** 一个新计划同时切换一个或多个模块选择版本
- **THEN** 底座只重新加载一次，之后仅导入新计划选择的版本

#### Scenario: 启停改变计划后重新加载
- **WHEN** 启用或停用操作成功改变选择模块集合
- **THEN** 底座重新加载并按新拓扑顺序恢复贡献

#### Scenario: 激活失败重新求解
- **WHEN** 模块激活失败使 Rust 阻止该版本并生成不同的兼容计划
- **THEN** 底座只自动刷新一次，随后按新计划加载；依赖失败模块不会在同一代次反复执行

#### Scenario: 无选择变化时不刷新
- **WHEN** 包只作为等待版本保存，或操作被依赖影响检查拒绝
- **THEN** 当前 WebView 与活动贡献保持不变，管理页面就地更新诊断
