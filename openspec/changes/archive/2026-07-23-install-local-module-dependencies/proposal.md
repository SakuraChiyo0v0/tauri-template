## Why

本地模块市场目前一次只能安装用户点选的单个 `.mtp`。当目标模块缺少必需依赖时，包虽然可以保存，但用户必须自己理解依赖诊断、查找对应文件并逐个安装，市场还不能完成一次完整的功能安装。

现在底座已经具备本地仓库扫描、版本求解、权限等待和单包恢复能力，适合把这些能力组合成可预览、可确认、失败不留半成品的依赖联动安装。

## What Changes

- 当前情况：安装目标模块前，用户需要手动安装它的所有必需依赖。
- 改造后：市场可以从同一个已授权本地仓库中寻找目标模块的传递必需依赖，并在写入前生成完整安装计划。
- 安装计划展示每个模块将执行的动作、选用版本、依赖关系、权限变化和不能继续的原因；用户确认前不修改现有模块。
- 用户确认后，底座按依赖顺序执行整份计划。任何包在最终校验、写入或计划提交阶段失败时，本次新增内容全部撤销，最后一个完整可用计划保持不变。
- 缺包、版本冲突、依赖循环、需要破坏已有依赖者或需要自动降级时，系统拒绝执行并给出结构化诊断。
- 可选依赖不会被自动安装；依赖包需要的新权限不会被自动批准，安装完成后仍按现有权限等待规则处理。
- 本次不增加联网下载、后台更新、跨仓库搜索、自动批准权限或远程市场。

## Capabilities

### New Capabilities

- `local-module-dependency-installation`: 定义从已授权本地仓库生成依赖闭包安装计划、用户确认、批量执行与失败恢复的行为。

### Modified Capabilities

- `local-module-market`: 市场页面从单包安装扩展为展示并确认目标模块及其必需依赖的完整安装计划。
- `runtime-module-dependency-resolution`: 依赖求解从仅考虑已安装版本扩展为在预览阶段同时考虑授权仓库中的候选包，且不得破坏当前可用组合。
- `module-package-lifecycle`: 包生命周期增加多包事务式安装，确保批量操作不会留下部分安装结果或部分计划切换。

## Impact

- 基座 Rust 模块仓库扫描、依赖求解和包存储边界需要增加“预览计划”和“确认执行”接口。
- Host SDK V6 在 V5 单包仓库 API 之上增加结构化安装计划与确认执行接口；V2–V5 行为保持不变。
- 独立 `local-module-market` 模块需要增加计划确认界面、冲突诊断和批量结果展示。
- `tauri-module-template` 的 SDK V5 类型、模拟宿主和测试需要同步更新。
- 现有单包安装、权限批准、激活失败恢复、回滚和旧版 SDK 行为保持兼容。
