## Context

当前底座可以从一个已授权本地目录扫描和安装单个 `.mtp`，也能在已安装版本中求解依赖组合。两者之间没有连接：仓库中的包不会成为依赖求解候选，市场模块也无法在一次操作中安装目标模块的传递必需依赖。

这项变更跨越公开 Host SDK、Rust 包存储与依赖求解、独立市场模块和模块模板。安全约束仍然是：模块看不到真实目录路径，不能任意读取包，不能绕过最终校验，也不能批准其他模块的权限。

## Goals / Non-Goals

**Goals:**

- 从目标 `.mtp` 出发，在同一个已授权仓库内计算传递必需依赖闭包。
- 在任何写入前返回确定、可解释的安装计划和权限变化。
- 用户确认后把整批包作为一个事务执行，失败或进程中断不留下部分安装结果。
- 保持当前可用模块组合优先，拒绝自动降级和会破坏现有依赖者的方案。
- 保持市场页面可拆卸，基座只提供通用仓库计划能力。

**Non-Goals:**

- 联网下载、远程索引、后台检查更新或跨目录搜索。
- 自动安装可选依赖、自动批准权限或自动停用现有模块。
- 解决不受信任 JavaScript 的沙箱隔离或包发布者身份问题。
- 把依赖计划界面写入基座模块管理页。

## Decisions

### 1. 新增 Host SDK V6，不改变 V5 单包接口

V6 在现有 `moduleRepository` 能力下增加“预览依赖计划”和“执行已确认计划”两个操作。市场模块升级到 V6；V2–V5 模块继续使用原有接口。

这样可以让模块仅凭 `sdkVersion` 判断计划 API 是否存在，避免把新方法悄悄加入 V5 后，模块在较旧的 V5 基座上运行时才发现方法缺失。模块仍需通过 `hostVersion` 声明具体底座版本范围。

备选方案是直接扩充 V5。它改动较少，但会让同一个 SDK 版本对应两套不同能力，因此不采用。

### 2. 计划由 Rust 生成并用短期不透明 ID 引用

市场提交目标包文件名后，Rust 重新扫描授权仓库、校验候选包并返回：

- 不透明 `planId`；
- 目标模块和依赖关系；
- 每个模块选用的版本与来源；
- `install`、`upgrade`、`keep` 或 `wait_permission` 等可展示动作；
- 权限摘要与结构化阻塞诊断。

`planId` 绑定调用模块会话、目录 grant、目标文件和候选包摘要，只在当前会话内短期有效。市场不能提交自己构造的文件路径或模块列表。

备选方案是把完整计划交还给市场，再由市场传回执行。这样容易被修改，也会扩大 Host SDK 输入面，因此不采用。

### 3. 执行前重新校验仓库快照

执行 `planId` 时，Rust 再次读取并完整校验计划涉及的每个包，同时比较文件摘要与关键仓库状态。如果包被替换、删除，grant 失效，已安装版本或活动计划发生变化，系统返回 `stale_plan`，不执行任何写入；市场必须刷新计划并让用户重新确认。

这比只依赖预览时的校验更能抵抗“预览后替换文件”的时间差问题。

### 4. 依赖求解采用最小扰动策略

目标文件的模块和版本固定。求解器随后只展开必需依赖：

1. 当前已选择版本满足范围时保持不变；
2. 已安装的其他兼容版本可满足时优先复用；
3. 仍缺失时，从同一仓库选择最高的兼容版本；
4. 当前版本不兼容时，只允许升级到能保持整张现有期望启用图可解的版本；
5. 需要降级、形成循环、缺包或破坏已有依赖者时返回阻塞诊断。

同一模块和版本存在多个文件时，只有内容摘要一致才能作为同一候选；内容不同则报告仓库冲突。相同输入必须产生相同计划和诊断顺序。

可选依赖不进入自动安装闭包，避免用户为非必要能力安装额外代码和权限。

### 5. 使用事务暂存、提交日志和恢复记录

执行时先把所有待写入包完整校验并展开到单个事务暂存目录，再写入一条包含计划代次和新版本目录的事务记录。只有全部版本准备完成后，才依次提升为不可变版本目录，并在最后原子提交新的全局计划。

- 全局计划提交前失败：删除本事务新建的版本目录并保留旧计划。
- 全局计划提交后失败：新计划已经是完整提交，启动恢复只清理事务记录和暂存目录。
- 进程中断：下次启动依据事务记录判断计划代次，回滚未提交的新版本或完成已提交事务的清理。

已存在的版本目录、数据库、权限批准和 grant 不属于事务删除范围。

### 6. 权限只展示和等待，不在批量安装中批准

计划汇总每个新增或升级模块的权限变化，但确认安装不等于确认权限。包写入后，权限扩大或首次请求原生能力的模块继续进入现有“等待权限”状态；其依赖者按现有依赖与激活规则等待。

这保留了“安装代码”和“授权系统能力”两个独立决定，也避免市场模块代替用户批准其他模块权限。

## Risks / Trade-offs

- **仓库包较多时预览变慢** → 继续限制顶层条目数量，只解析 `.mtp`，对单次预览缓存已校验摘要，并为包数量与展开大小保留现有限制。
- **事务恢复逻辑比单包安装复杂** → 事务记录保持最小，只记录事务 ID、基线计划代次和本事务新建目录；增加进程中断恢复测试。
- **最高兼容版本不一定是用户偏好的版本** → 首版强调确定性和最少交互；计划会明确展示选择结果，用户可以取消并整理仓库。自定义版本选择留待后续变更。
- **权限等待可能让“安装成功”看起来没有立即运行** → 市场结果明确区分“已安装并运行”“已安装等待权限”和“已安装等待依赖”。
- **V6 增加模板与测试维护量** → V5 接口保持不动，V6 只增加仓库计划类型和模拟宿主路径，避免影响无关模块。

## Migration Plan

1. 基座先发布同时支持 V2–V6 的 Host SDK 和 Rust 计划接口。
2. 模块模板增加 V6 类型、模拟计划和打包测试，默认新模块可按需选择 V6。
3. 本地市场模块提高 `sdkVersion` 与 `hostVersion`，改用预览和执行接口。
4. 旧版市场模块仍可使用 V5 单包安装；升级市场模块后才出现依赖联动界面。
5. 如果发布后需要回滚，停用或回滚市场模块即可恢复单包交互；基座保留的 V6 加法接口不会改变旧模块行为。

## Open Questions

当前首版没有阻塞实施的问题。后续可单独讨论用户手动选择依赖版本、跨仓库搜索和签名包信任策略。
