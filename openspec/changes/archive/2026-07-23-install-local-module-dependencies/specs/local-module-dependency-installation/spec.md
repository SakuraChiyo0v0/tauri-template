## ADDED Requirements

### Requirement: 系统必须在写入前生成完整依赖安装计划
Host SDK V6 SHALL 允许已获批模块仓库能力的模块为一个顶层目标 `.mtp` 请求安装计划。系统 MUST 从同一授权仓库中展开传递必需依赖，返回每个模块的选用版本、依赖关系、安装动作、权限摘要和结构化诊断，并且在用户确认前不得修改安装目录、权限或活动计划。

#### Scenario: 目标模块存在可满足的传递依赖
- **WHEN** 目标模块 A 依赖 B，B 依赖 C，且同一授权仓库包含兼容的 B 与 C
- **THEN** 系统 SHALL 返回包含 C、B、A 的完整计划及 provider 优先顺序，不写入任何模块版本

#### Scenario: 预览缺少必需依赖
- **WHEN** 目标模块或其传递依赖需要仓库和已安装版本中都不存在的模块
- **THEN** 系统 SHALL 返回包含模块 ID 与版本范围的阻塞诊断，且计划不得执行

#### Scenario: 可选依赖存在于仓库
- **WHEN** 目标模块声明一个尚未安装但仓库中存在的可选依赖
- **THEN** 系统 MUST NOT 把该可选依赖加入自动安装动作，目标模块仍可继续规划

### Requirement: 依赖计划必须确定且优先保持现有可用组合
系统 MUST 固定用户选择的目标包版本，并在满足整张期望启用依赖图的前提下优先保持当前选择、复用已安装兼容版本，再从仓库选择最高兼容版本。系统 MUST 拒绝自动降级、依赖循环、内容冲突或会破坏现有依赖者的方案。

#### Scenario: 当前依赖版本已经兼容
- **WHEN** 目标模块需要 B `^1.0.0` 且当前已选择 B `1.2.0`
- **THEN** 计划 SHALL 保持 B `1.2.0`，不得仅因仓库存在更高兼容版本而升级 B

#### Scenario: 依赖需要兼容升级
- **WHEN** 当前 B 不满足目标范围，仓库中的更高版本同时满足目标模块和所有现有依赖者
- **THEN** 计划 SHALL 把 B 标记为升级并展示受影响关系

#### Scenario: 只能通过降级满足目标
- **WHEN** 目标模块只能通过降低当前依赖版本才能满足
- **THEN** 系统 SHALL 阻止计划并返回自动降级被拒绝的结构化原因

#### Scenario: 相同版本文件内容冲突
- **WHEN** 仓库中同一模块 ID 与版本出现摘要不同的多个包
- **THEN** 系统 SHALL 报告仓库冲突，不任意选择其中一个文件

### Requirement: 已确认计划必须防止过期或被篡改
系统 MUST 以绑定当前模块会话、目录 grant、目标文件、候选包摘要和基线计划代次的不透明 `planId` 引用预览结果。执行时 MUST 重新校验全部相关包和仓库状态；任何关键输入变化都必须使计划失效且不得产生写入。

#### Scenario: 预览后包被替换
- **WHEN** 用户确认前，计划涉及的 `.mtp` 被删除、替换或内容发生变化
- **THEN** 执行 SHALL 返回 `stale_plan`，现有版本与活动计划保持不变

#### Scenario: 其他操作改变活动计划
- **WHEN** 生成计划后另一个模块操作提交了新的全局计划代次
- **THEN** 原 `planId` SHALL 失效，市场必须重新预览并取得用户确认

#### Scenario: 其他模块尝试执行计划
- **WHEN** 非创建者会话提交该 `planId`
- **THEN** 系统 MUST 拒绝请求且不得暴露计划内容或读取仓库包

### Requirement: 批量安装不得隐式批准模块权限
系统 SHALL 在计划中展示所有待安装或升级模块的权限摘要和权限变化，但确认安装 MUST NOT 创建或扩大这些模块的权限批准。安装后的模块 MUST 继续遵循现有权限等待与依赖激活规则。

#### Scenario: 依赖模块首次请求原生权限
- **WHEN** 用户确认包含一个需要原生权限的新依赖模块的安装计划
- **THEN** 包可以完成安装，但该依赖模块 SHALL 显示等待权限，系统不得替用户批准

#### Scenario: 依赖升级扩大权限
- **WHEN** 计划包含权限范围扩大的依赖版本升级
- **THEN** 计划 SHALL 明确标记权限变化，执行后新版本仍等待单独批准，旧批准不得自动扩大
