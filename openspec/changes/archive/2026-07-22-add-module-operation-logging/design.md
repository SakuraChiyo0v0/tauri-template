## Context

基座已经把 `hostSdk.logger` 接入统一日志控制台，并自动以模块 ID 标记来源。当前缺口在模块侧：模板没有规定操作日志，三个参考模块也只记录激活和停用，因此日志控制台无法解释用户刚才执行的操作及其结果。

这次改动横跨独立模块模板和三个本地模块，但不改变 Host SDK、日志存储或日志页面。

## Goals / Non-Goals

**Goals:**

- 让新模块从模板开始就记录少量、高价值的生命周期、成功操作和失败日志。
- 让三个参考模块的主要用户操作可以在基座日志控制台中追踪。
- 保证日志不包含正文、凭据、完整 URL、文件路径或其他不必要的用户数据。
- 用聚焦测试固定日志级别、操作类型和敏感信息边界。

**Non-Goals:**

- 不增加新的 Host SDK 日志接口、结构化日志协议或日志持久化能力。
- 不记录每次渲染、输入和数据库查询等低价值细节。
- 不改变日志控制台界面、全局日志级别或模块业务行为。

## Decisions

### 1. 模块只通过 Host SDK 写日志

模块使用 `hostSdk.logger`，由基座自动添加模块 ID 来源。不会使用 `console.log`，也不会直接调用 Tauri 日志插件，因为这些路径无法保证统一收集和模块归属。

### 2. 使用稳定、简短的英文事件消息

日志消息使用稳定英文短句，必要时只附加非敏感结果，例如数量、记录 ID、触发来源或退出码。这样无需修改现有纯文本日志 ABI，也便于搜索和测试。

曾考虑把对象作为结构化字段传给日志接口，但当前公共 ABI 只接受字符串；本次不为日志规范扩大 Host SDK。

### 3. 以结果严重程度选择日志级别

- `info`：激活、停用，以及用户明确发起并成功完成的关键操作。
- `warn`：输入无效、依赖暂不可用、用户取消等无需崩溃且可恢复的结果。
- `error`：数据库、服务或原生能力调用失败。
- `debug` 与 `trace` 不承担用户操作的唯一记录，因为基座默认阈值为 `info`。

### 4. 默认最小化日志数据

日志不得包含笔记标题或正文、完整 URL、凭据、授权内容、文件系统路径、服务负载或原始错误对象。错误日志只保留操作名称和经过清理的错误类别或固定说明；参考模块不拼接潜在用户输入。

### 5. 模板规范与可执行示例同时更新

`AGENTS.md` 约束后续开发者和 AI；README 解释可观察行为；示例模块和测试展示正确写法。三个现有模块按同一规则补齐，防止规范只停留在文档中。

## Risks / Trade-offs

- [纯文本消息不如结构化日志易于聚合] → 使用稳定事件短句和有限的 `key=value` 信息，未来若扩展 ABI 再迁移。
- [日志过多会降低可读性] → 只记录用户明确操作的最终结果，不记录渲染、订阅和内部查询。
- [错误详情过少可能降低排查效率] → 在不暴露用户数据的前提下保留操作名和安全的错误类型；深层原生错误仍由基座运行时日志承担。
- [多个本地仓库需要保持一致] → 由模板规范作为基线，并用每个模块的聚焦测试验证关键事件。
