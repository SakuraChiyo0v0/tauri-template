## Context

底座版本同时存在于 npm 包、Rust crate、Tauri 配置和 Cargo 锁文件中。运行时模块安装服务使用 Rust crate 版本校验 `hostVersion`，前端 Host SDK 使用 npm 包版本报告底座版本，因此这些位置不一致会直接造成兼容性判断错误。

项目仍处于 `0.x` 阶段。本次依赖解析属于向后兼容的新能力，按语义化版本提升 minor；未来破坏 Host SDK 或模块包协议时也必须明确提升兼容边界。

## Goals / Non-Goals

**Goals:**

- 用一个无额外依赖的命令同步所有项目版本位置。
- 在写入前检查当前版本一致性、目标版本格式和对应 changelog 条目。
- 支持 `major`、`minor`、`patch` 和明确版本号四种输入。
- 为发布前自动化提供只读的一致性检查命令。

**Non-Goals:**

- 不自动创建 Git commit、tag、GitHub Release 或推送远端。
- 不自动从提交信息生成 changelog。
- 不管理独立 `.mtp` 模块自身的版本。

## Decisions

### 使用仓库内 Node 脚本，不引入发布框架

新增 `scripts/version.mjs`，直接读取并更新现有四个版本位置。项目已经依赖 Node 和 pnpm，因此不增加 Changesets、release-it 或 cargo-release 等工具依赖，模板复制后也没有额外配置成本。

备选方案是引入完整发布框架，但当前只有一个桌面应用包，没有多包发布、npm 发布或自动 GitHub Release 需求，收益不足以抵消配置复杂度。

### 变更记录人工维护，脚本负责强制校验

`CHANGELOG.md` 使用 Keep a Changelog 风格，保留 `Unreleased` 并为每个正式版本提供日期。版本升级命令只有在目标版本条目已经存在时才写入，从流程上保证版本号与变更记录同步。

自动从 Git 提交生成 changelog 会把内部重构和修复细节直接暴露给使用者，也依赖严格的提交规范，因此首版不采用。

### 发布命令只修改版本文件

脚本不执行测试、提交或推送。发布者先维护 changelog，再运行版本升级，最后执行仓库已有验证命令并自行决定 Git 操作。这保留了清晰的权限边界，也避免脚本在失败时产生不可预期的外部状态。

## Risks / Trade-offs

- [人工 changelog 可能遗漏变化] → 版本命令要求目标版本条目存在，并在 README 中记录发布顺序。
- [多个文件写入中断可能造成暂时不一致] → `version:check` 会明确列出不一致位置；脚本在写入前完成全部解析和校验。
- [提升到 `0.2.0` 后，声明 `<0.2.0` 的旧模块不兼容] → 更新示例模块兼容范围；真实模块必须按其声明发布兼容版本，底座不绕过清单约束。
