## Context

运行时模块在共享 WebView 中执行，当前 Host SDK V1 只提供日志、设置和主题；并行的 `runtime-module-sqlite-storage` 变更将增加 Host SDK V2 隔离数据库。模块不能直接调用原始 Tauri invoke，也不能依赖编译期为某个业务模块写死的 Tauri 权限。文件、进程、注册表、托盘和全局快捷键都涉及用户数据或系统级资源，必须由底座统一代理、解释和回收。

模块仍属于“用户主动安装的本地受信代码”。本设计通过最小授权和不透明会话令牌降低误用与模块间串用，不声称共享 WebView 能对恶意 JavaScript 提供进程级隔离。

## Goals / Non-Goals

**Goals:**

- 让模块在不获得原始 Tauri API 的情况下完成常见桌面工具工作流。
- 安装前展示清单声明的原生能力，新增权限的升级必须重新确认。
- 使用与模块、版本和当前激活计划绑定的不透明会话令牌校验每次原生调用。
- 为文件、进程和注册表定义可验证的最小访问范围，并为托盘和快捷键提供共享资源仲裁。
- 保持 Host SDK V1/V2 模块可运行；V3 在 V2 数据库接口基础上增加原生能力。
- 停用、卸载、回滚、激活失败和权限撤销时释放会话、托盘项和快捷键。

**Non-Goals:**

- 不提供任意 Tauri invoke、任意 Shell 字符串、管理员提权或系统服务管理。
- 不允许模块携带和动态加载 Rust、DLL 或其他原生扩展。
- 不承诺对同一 WebView 中的恶意模块形成强安全沙箱。
- 不提供后台子进程、交互式终端、进程注入或无限输出流。
- 不允许注册表 HKLM 写入，也不在非 Windows 平台模拟注册表。
- 不允许模块替换应用托盘、抢占其他模块快捷键或绕过用户授权。

## Decisions

### 清单声明与运行授权分离

V3 模块清单增加 `nativeCapabilities`。清单只描述模块可能使用的最小能力和静态贡献，不等于已经授权。Rust 安装状态另外保存用户批准的声明摘要和权限指纹；新安装或升级扩大声明时，包可以落盘，但不得进入激活计划，状态为等待权限确认。

首版采用全量批准：模块声明的原生权限全部批准后才能激活，不提供“只批准一部分仍启动”。这样可以明确模块启动条件并避免每个模块自行解释半授权状态。用户之后撤销任一权限时，底座先停用模块并释放运行资源，再保存撤销状态。

备选的逐项可选权限更灵活，但会引入 required/optional 组合、更多诊断和不可预测的启动行为，留待出现真实用例后扩展。

### V3 使用不透明激活会话令牌

加载器在激活 V3 模块前向 Rust 创建会话。Rust只为当前计划选中的模块版本签发随机令牌，并把批准权限快照绑定到令牌。Host SDK 闭包只传令牌和操作参数，不传可由模块伪造的路径或授权模块 ID。

重新加载、停用、卸载、回滚、激活失败或撤销权限时，旧令牌失效。令牌降低普通模块串用其他模块权限的风险；同一 WebView 的威胁模型仍要求模块是用户选择的受信代码。

### 文件访问采用私有根目录与用户授权句柄

每个模块始终只能通过 V3 文件 API 访问底座分配的 `runtime-module-data/<module-id>/files` 私有根目录，参数为安全相对路径，拒绝绝对路径、父目录跳转和符号链接逃逸。

私有根目录之外的文件或目录必须由底座选择器返回不透明 grant ID。Rust 持久化规范化路径、文件类型和读/写权限；Host SDK 后续只接受 grant ID。授权可以在模块管理页撤销。读取、写入、目录枚举和二进制传输设置固定大小上限；写入使用临时文件后替换，避免部分写入。

### 进程能力不经过 Shell

`openUrl` 只允许清单声明并由用户批准的 URL scheme。`openPath` 和 `revealInFolder` 只接受文件授权句柄。

执行程序必须先由用户在文件选择器中选择可执行文件并生成 executable grant。Rust 使用直接进程 API启动，不拼接 Shell 字符串，不提权、不接受 stdin，并使用最小环境。调用限制参数数量、参数长度、超时和 stdout/stderr 总量；超时后终止子进程。首版只有有界 `run`，没有后台 `spawn`。

### 注册表按平台和 hive 限制

清单声明注册表 scope：hive、键前缀和 `read`/`read-write`。Windows 上仅允许 HKCU 读写和 HKLM 只读；模块不能申请其他 hive、远程注册表或 HKLM 写入。每次操作把目标子键规范化并确认仍位于批准前缀下。值支持字符串、DWORD、QWORD、二进制和多字符串；删除仅限 HKCU scope 内的值，不递归删除整个键树。

非 Windows 平台保留同一 TypeScript 接口并返回结构化 `unsupported_platform`，让跨平台模块能够显式降级。

### 托盘是声明式共享贡献

应用只有一个由底座持有的托盘。模块在清单中声明稳定菜单 ID、标签、类型和顺序；底座按模块分组并命名空间化实际 ID。Host SDK 允许订阅本模块事件，以及更新已声明项目的标签、启用和选中状态，不能创建清单外项目、移除底座项目或替换托盘图标。

没有模块贡献时底座不必创建额外托盘；存在贡献时保留“显示主窗口”和“退出”底座项目。模块失活时只移除自己的分组。

### 快捷键由底座仲裁并允许用户覆盖

模块清单声明快捷键 ID、用途说明和默认 accelerator。批准后底座尝试注册；冲突时不抢占现有所有者，记录结构化冲突并让模块继续运行。用户可以在模块管理页禁用或重新绑定，成功绑定后持久化覆盖值。

快捷键事件只发送给拥有该 ID 的活动模块会话。停用、回滚、卸载、权限撤销和会话失效都必须注销。底座启动时按激活计划拓扑顺序确定注册顺序，使冲突结果可复现。

### Rust 自建代理而不是向模块暴露插件 API

底座可以内部使用 Tauri dialog、opener、global-shortcut、tray API和 Windows 注册表 crate，但前端运行时模块只能调用自有的 `native_capabilities` Tauri commands。Rust 命令首先校验会话，再校验声明、用户授权和资源归属。

这样比直接开放 `tauri-plugin-fs`、`tauri-plugin-shell` 或原始 invoke 多一层代码，但能固定路径、输出上限和生命周期语义，也不会让动态模块扩大应用编译期 Tauri capability。

### 与 SQLite V2 串行集成

本 change 的规格和独立 Rust 能力代理可以并行开发，但 Host SDK V3 类型、清单 SDK 版本和加载器改动必须在 `runtime-module-sqlite-storage` 合入后应用。V3 是 V2 的严格超集：数据库成员保持不变，再增加 `filesystem`、`process`、`registry`、`tray` 和 `shortcuts`。

## Risks / Trade-offs

- [同一 WebView 中恶意模块可能尝试调用内部 bridge] → 不公开命令名和令牌、绑定激活会话并保持受信模块威胁模型；强隔离另立进程/WebView 规格。
- [权限清单过宽使用户难以判断风险] → 使用结构化摘要、禁止通配根路径、安装前展示具体 scheme、注册表前缀和托盘/快捷键贡献。
- [持久文件 grant 指向移动或替换后的文件] → 每次访问重新规范化并检查类型；失效 grant 返回明确错误，可由用户重新选择。
- [执行用户选择程序仍可能产生副作用] → 无 Shell、无提权、最小环境、有界输出和超时；安装页明确标记为高风险能力。
- [Windows 注册表导致跨平台差异] → 统一结构化错误，清单和 UI 标明 Windows 专属，测试覆盖 Windows 实现和非 Windows拒绝路径。
- [托盘和快捷键是全局共享资源] → 底座统一所有权、命名空间和确定性冲突处理，模块失活时强制清理。
- [与 SQLite V2 修改同一 Host SDK 文件] → 先实现独立代理并等待 V2 合入，再集中完成 V3 衔接和回归测试。

## Migration Plan

1. 先实现不依赖 Host SDK 的权限模型、会话管理和五类 Rust 代理测试。
2. 等 SQLite V2 合入后，将清单和前端类型扩展到 SDK V3，并保持 V1/V2 判别联合。
3. 接入安装权限确认、权限状态持久化和模块管理界面。
4. 接入托盘/快捷键生命周期与真实 Tauri 插件，再更新独立模块模板。
5. 使用真实 V3 `.mtp` 做权限批准、文件/进程/注册表、托盘、快捷键、撤销和重启冒烟。

回滚到底座旧版本时，V3 模块应因 SDK/hostVersion 不兼容而不激活；已保存文件 grant、权限决定和私有数据保留，重新升级后可以继续使用。

## Open Questions

- Linux/macOS 是否需要提供各自的设置存储替代 Windows 注册表，留到出现跨平台模块需求时决定。
- 外部文件 grant 的默认有效期首版设为持久，未来可增加“仅本次会话”选项。
- 后台进程和进程事件流是否值得单独能力，需要真实模块用例后再设计。
