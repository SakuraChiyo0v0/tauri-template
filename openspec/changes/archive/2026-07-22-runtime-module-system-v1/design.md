## Context

当前 `FeatureRegistry` 只支持在启动前静态注册源码模块，页面直接携带 React 组件，Rust 插件也在应用构建时固定注册。这种结构适合作为模板起点，但不能在已安装的底座中替换单个业务功能。现阶段代码量仍小，应该先重构宿主边界，再让后续模块依赖稳定协议。

V1 面向用户自己构建和主动选择的受信模块。模块代码可以运行在主 WebView 中，但只能依赖公开宿主 SDK；模块包不能直接加入新的 Rust 命令、Tauri 插件或系统权限。

## Goals / Non-Goals

**Goals:**

- 不重新安装底座即可安装或升级前端业务模块。
- 模块更新失败时保留或恢复上一个可用版本。
- 运行时模块继续使用统一导航、设置、主题和日志体验。
- 把模块格式、兼容性和生命周期变成可测试的稳定契约。
- 重构当前注册表，使内置模块与运行时模块都通过同一聚合层驱动 UI。
- 为以后接入远程版本源、签名和模块开发工具保留清晰扩展点。

**Non-Goals:**

- V1 不提供远程模块商店、后台自动下载或更新通知服务。
- V1 不运行来源不明的第三方代码，也不建立公钥信任链。
- V1 不动态加载 Rust 动态库、Tauri 插件或新增系统权限。
- V1 不保证安装后无刷新热替换；升级、回滚和卸载后允许重新加载 WebView。
- V1 不提供跨底座大版本的自动数据迁移。

## Decisions

### 1. 使用 `.mtp` ZIP 包和严格清单

模块包是扩展名为 `.mtp` 的 ZIP 文件，根目录必须包含 `manifest.json` 和一个清单声明的单文件 ESM 入口。清单包含：

- `schemaVersion`
- `id`、`name`、`description`、`version`
- `hostVersion` 语义版本范围和 `sdkVersion`
- `entry`
- 导航贡献及其自定义元素标签
- 仅含 `switch`、`select` 的声明式设置贡献

Rust 安装服务限制包体大小、解压后大小、文件数量和相对路径，并拒绝绝对路径、父目录跳转、符号链接、重复文件、非法标识和不兼容版本。安装时记录原包 SHA-256。

选择 ZIP 是为了得到单文件分发和可重复校验。V1 不接受任意目录作为正式安装源，避免目录在校验与复制之间发生变化。

### 2. Rust 拥有磁盘状态，前端拥有 UI 激活

Rust 命令负责列出、校验、安装、切换、回滚和卸载模块版本。模块目录位于应用数据目录：

```text
modules/<module-id>/
├─ state.json
└─ versions/<version>/
   ├─ manifest.json
   ├─ index.js
   └─ assets/...
```

状态文件记录当前版本、上一版本、文件哈希和最近激活错误，并通过同目录临时文件和原子替换写入。安装先完整写入暂存目录，校验成功后才移动到版本目录；任何失败都不得修改活动版本。

前端读取活动模块清单并注册贡献。模块启用时，前端读取入口源码，使用 Blob URL 动态导入单文件 ESM，然后调用其 `activate(hostSdk)`。这种职责划分避免 Rust 理解前端框架，同时避免前端直接写入安装目录。

### 3. 运行时页面使用 Web Components 边界

运行时模块入口负责定义清单声明的自定义元素。宿主使用通用 React 包装页渲染这些元素，而不是要求模块导出宿主内部的 React 组件。

选择 Web Components 是为了避免底座与模块打包出两个 React 实例所造成的 Hook/Context 冲突。模块可以自行选择 React、Preact 或原生 DOM，并把依赖打入单文件入口。主题通过继承的 CSS 自定义属性同步，公共视觉规范由模块 SDK 提供。

备选方案是共享 React 单例和 Module Federation，但它会把构建工具、React 版本和运行时加载器变成模块 ABI，V1 不采用。

### 4. 导航和设置由清单声明，代码只实现功能

底座无需执行模块代码即可展示模块信息和设置结构。导航标题、说明、分组、顺序和元素标签来自清单；设置只允许底座能安全渲染的 `switch` 与 `select`。

这延续现有“模块注册贡献、共享页面负责渲染”的方向，并避免模块任意替换设置页面。更复杂的设置 UI 应作为模块自己的导航页面实现。

### 5. 重构注册表为可观察的动态聚合层

注册表将支持：

- 标记模块来源为 `builtin` 或 `runtime`
- 运行时注册和注销
- 订阅快照，使模块变化自动刷新侧边栏、设置和模块管理器
- 独立执行启用、停用和清理生命周期
- 防止模块 ID、导航 ID 和自定义元素标签冲突

内置系统模块和日志模块继续源码注册；运行时模块由启动引导器发现。应用外壳仍不直接导入业务页面。

### 6. 更新后刷新，激活失败自动回滚一次

安装更高兼容版本后，Rust 将新版本设为活动版本并保留原活动版本为上一版本。前端随后重新加载 WebView，让旧模块代码和自定义元素定义彻底释放。

如果新入口导入、`activate` 或声明元素验证失败：

- 有上一版本时，前端报告失败，Rust 原子恢复上一版本并记录错误，然后只自动刷新一次。
- 没有上一版本时，Rust 将该版本标记为激活失败；底座继续启动，但不注册其页面，并在模块管理器显示错误。

刷新边界比尝试在同一 JS 上下文卸载模块更可靠，也为将来的后台下载与下次启动激活留下空间。

### 7. 首版 SDK 是显式对象，不暴露内部模块

`activate(hostSdk)` 获得带版本的最小 SDK：

- 模块身份与宿主版本
- 分级日志
- 读取和写入本模块命名空间设置
- 订阅主题变化

模块不得导入 `src/core` 或调用未公开的注册表对象。新增 SDK 能力必须提升 SDK 版本并声明底座兼容范围。

### 8. 本地文件选择使用官方对话框插件

模块管理页面通过 Tauri 官方 dialog 插件选择 `.mtp` 文件，并调用受限 Rust 安装命令。浏览器开发预览不伪造安装结果，而是明确显示安装需要在 Tauri 运行环境中执行。

## Risks / Trade-offs

- **[受信模块在主 WebView 中拥有较高权限]** → V1 只接受用户主动选择的本地包，严格校验结构并记录哈希；远程分发前必须新增签名和权限代理。
- **[Web Components 与现有 React 组件库不能直接共享实例]** → 通过 CSS 主题令牌和独立模块 SDK 共享视觉规范；避免不稳定的 React ABI。
- **[单文件 ESM 限制大型模块和资源引用]** → V1 允许包内静态资源但入口必须由模块构建工具打成单文件；后续可通过只读模块资源协议扩展资源加载。
- **[更新需要刷新 WebView]** → 在操作前保存底座设置，刷新后恢复导航；换取确定的代码卸载和回滚行为。
- **[安装流程涉及压缩包攻击面]** → Rust 端实施包体、展开大小、文件数、路径和文件类型限制，所有目标路径都从已验证标识构造。
- **[底座 SDK 过早膨胀]** → V1 只提供日志、设置和主题，新增原生能力必须经过新的规格变更。

## Migration Plan

1. 保留现有内置系统和日志模块，先重构注册表但不改变其用户行为。
2. 引入 Rust 模块存储及命令，并使用临时应用数据目录完成安装、升级、失败不切换和回滚测试。
3. 引入运行时启动引导器和 Web Component 页面包装器。
4. 扩展模块管理页面，加入本地安装、升级、回滚和卸载入口。
5. 提供一个仅用于测试和文档的最小示例模块源码及打包说明，不把演示模块预装进底座。
6. 若发布后需要回退，可移除运行时引导器和管理入口；内置模块仍能独立运行，已安装包保留在应用数据目录但不会被加载。

## Open Questions

- 远程模块源、签名密钥和权限模型将在后续变更中决定；它们不阻塞本地受信模块 V1。
- 模块独立数据目录与数据库 API 将随第一个确有持久化需求的业务模块设计，不在 V1 提前扩大 SDK。
