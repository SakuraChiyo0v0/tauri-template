## Why

当前模块必须在构建时静态导入，更新任一业务功能都需要重新构建和安装整个桌面程序。项目需要把现有模板升级为稳定底座，使受信任的前端功能模块可以独立安装、升级、停用和回滚，同时继续复用底座的导航、设置、主题与日志能力。

## What Changes

- 引入带版本和兼容性声明的本地模块包格式。
- 在应用数据目录中保存已安装模块及其多个版本，并以原子方式切换当前活动版本。
- 在模块管理页面提供本地安装、升级、停用、卸载和回滚操作及明确结果。
- 运行时发现并加载活动模块，使模块不再需要写死到共享应用外壳中。
- 为运行时模块提供稳定的宿主 SDK，用于注册页面、设置项、日志和主题兼容界面。
- 安装前校验包结构、模块标识、语义版本、底座兼容范围和文件完整性；失败时不得破坏当前版本。
- 新版本激活或健康检查失败时恢复上一活动版本。
- 保留现有源码模块机制，用于底座内置的系统模块和日志基础设施。
- V1 只接受用户主动选择的本地受信模块包，不包含远程商店、自动下载、第三方签名信任链或动态 Rust/Tauri 插件。

## Capabilities

### New Capabilities

- `module-package-lifecycle`: 定义本地模块包的发现、校验、安装、升级、停用、卸载、版本保留和失败回滚行为。
- `runtime-module-contributions`: 定义运行时模块如何被加载，并向底座贡献页面、设置、主题兼容界面和生命周期状态。

## Impact

- 前端模块注册表将同时聚合内置源码模块和运行时模块贡献。
- Tauri 后端将新增受限的模块目录、包读取、安装和原子版本切换命令。
- 模块管理页面将从单纯启停内置模块扩展为管理已安装运行时模块。
- 将新增模块包清单、宿主 SDK 版本和底座兼容性契约。
- 安装流程会接触本地文件系统，因此必须限制解包路径、包大小、文件类型和目标目录。
- 动态原生能力仍由底座发布；需要新增 Rust 命令或 Tauri 插件时继续更新整个应用。
