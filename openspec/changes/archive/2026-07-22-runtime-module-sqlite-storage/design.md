## Context

运行时模块当前在同一 WebView 中执行，只能通过 Host SDK V1 使用日志、模块私有设置和主题。设置保存在 `localStorage`，适合少量配置，但不适合结构化数据、迁移和事务。Rust 模块存储只管理 `.mtp` 代码、清单和激活计划；卸载会删除代码目录，因此业务数据不能放在现有模块目录中。

运行时模块是用户主动安装的本地受信代码。Host SDK 边界用于提供稳定接口和防止模块因路径或 SQL 功能误伤其他数据，不构成对同一 WebView 中恶意 JavaScript 的强安全沙箱。

## Goals / Non-Goals

**Goals:**

- 为每个运行时模块提供持久、独立、可统计和可清理的 SQLite 数据库。
- 通过 Host SDK V2 提供参数化查询、执行、事务和 schema 版本管理。
- 保持现有 Host SDK V1 模块、V1 `.mtp` 包和安装状态可用。
- 由 Rust 决定数据库路径并拒绝跨数据库附加、扩展加载和危险 PRAGMA。
- 模块卸载后保留数据，并让用户能够识别和清理已卸载模块留下的数据。

**Non-Goals:**

- 不允许模块直接访问其他模块数据库或底座公共数据库。
- 不实现模块服务注册、跨模块 RPC、共享表或跨库外键。
- 不迁移现有 `localStorage` 设置，也不把 SQLite 作为日志存储。
- 不在首版实现云同步、加密数据库、配额或自动数据备份。
- 不把运行时模块提升为对抗恶意代码的进程级沙箱。

## Decisions

### 使用 `rusqlite` 由底座管理 SQLite

Rust 底座直接使用 `rusqlite` 打开数据库，而不是安装向 WebView 暴露通用连接字符串的 SQL 插件。数据库管理器可以固定数据根目录、安装 SQLite authorizer、限制语句和结果规模，并为测试注入临时目录。

备选的 `tauri-plugin-sql` 集成更快，但它的通用连接模型会扩大 Host SDK 和权限面，也更难保证模块只能访问自己的文件。

### 数据库与模块代码分离

数据库位于应用数据目录的 `runtime-module-data/<module-id>/data.sqlite3`。模块 ID 必须先通过现有清单规则校验，数据库 API 不接受文件路径。代码升级、回滚和普通卸载不会移动或删除该目录。

数据清理只允许删除精确模块数据目录中的数据库及 SQLite sidecar 文件。已启用模块必须先停用；已卸载模块留下的数据通过独立数据清单展示和清理。

### Host SDK 按模块清单协商 V1/V2

清单 `sdkVersion` 接受 `1` 或 `2`。加载 V1 模块时继续传入字面量 `sdkVersion: 1` 且不包含数据库成员；加载 V2 模块时传入 `sdkVersion: 2` 和 `database`。底座不把 V2 对象强塞给 V1 模块，避免已有模块的严格版本检查失败。

V2 数据库接口提供：

- `execute(sql, params)`：执行一条写入或 DDL 语句并返回影响行数和最后插入 ID。
- `select(sql, params)`：只执行 SQLite 判定为只读的单条语句并返回 JSON 兼容行。
- `transaction(statements)`：在一个事务中原子执行一组参数化写入或 DDL；任一步失败则全部回滚。
- `getUserVersion()` / `setUserVersion(version)`：由底座安全管理 schema 版本，不向模块开放任意 PRAGMA。

参数和值首版支持 `null`、布尔值、字符串、有限且不超出 JavaScript 安全整数范围的数字，以及以数字数组表示的二进制数据。语句、参数数量、返回行数和序列化结果设置固定上限，超限返回结构化错误。

### SQL 安全由 SQLite authorizer 和单语句执行共同保证

每次连接在执行模块 SQL 前启用外键，并安装 authorizer。底座拒绝 `ATTACH`、`DETACH`、任意 `PRAGMA`、扩展加载以及非 `main`/`temp` 数据库访问；`select` 还要求 prepared statement 为只读。接口不提供脚本批量执行，事务只能包含逐条校验的语句。

这能防止正常模块因为错误 SQL 越过自己的数据库文件。恶意模块仍属于现有“用户选择的受信包”威胁模型；若未来需要强隔离，必须引入独立 WebView 或进程边界。

### 数据统计使用独立清单 API

Rust 返回所有数据库目录的模块 ID、主文件与 sidecar 总字节数，以及该模块当前是否仍安装。前端将已安装模块的用量合并到模块卡片，并为已卸载但仍有数据的模块显示“保留的数据”条目。

清理操作返回最新数据清单，不改变模块激活计划，因此无需刷新整个 WebView。

## Risks / Trade-offs

- [每个模块有少量空库和 WAL sidecar 开销] → SQLite 引擎只打包一次；界面展示实际字节数，首版不为极小空间收益共享文件。
- [原始 SQL API容易扩大能力面] → 固定模块路径、单语句 prepared API、authorizer、参数和结果上限，并禁止任意 PRAGMA/ATTACH。
- [V1/V2 类型分支增加加载器复杂度] → 使用判别联合类型和按清单构造 SDK，保留针对两种版本的契约测试。
- [卸载保留数据产生孤立文件] → 数据清单独立于安装清单，模块管理器持续显示并允许显式清理。
- [同一 WebView 不是恶意模块安全边界] → 文档继续限定本地受信模块；更强沙箱另立规格，不在 SQLite 变更中伪造保证。
- [跨模块不能直接 SQL JOIN] → 后续以 provider 拥有数据、消费者调用版本化服务的方式共享，避免表结构耦合。

## Migration Plan

1. 引入 SQLite 管理器和纯 Rust 隔离、安全、事务与清理测试。
2. 注册数据库命令和数据清单命令，保持现有模块安装目录不变。
3. 扩展清单验证、前端类型和加载器以协商 Host SDK V1/V2。
4. 更新模块管理页、独立模块模板和真实 V2 模块冒烟。
5. 现有 V1 模块无需迁移；首次 V2 数据访问时按需创建数据库。

若底座回滚到不支持 V2 的版本，V2 模块会按现有 host/sdk 兼容校验拒绝安装或激活，已创建数据库保持原样，重新升级后可继续使用。

## Open Questions

- 模块间服务层的调用协议、权限和版本发现将在后续 change 中决定。
- 数据配额、导出/导入和加密需求等出现真实用例后再扩展，不在 V2 首版预设。
