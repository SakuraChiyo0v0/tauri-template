## MODIFIED Requirements

### Requirement: 用户必须能够管理模块生命周期
模块管理界面 SHALL 展示每个运行时模块的选择版本、可用版本、期望启用状态、依赖、依赖者、最近错误和数据库占用，并提供与当前依赖图相符的启用、停用、回滚和卸载操作。系统 MUST 在提交操作前分析直接与间接影响，且不得静默级联修改其他模块。卸载 MUST 删除模块代码和安装状态但默认保留模块数据库；已卸载模块的保留数据 MUST 继续可见并可由用户显式清理。

#### Scenario: 启用等待中的模块
- **WHEN** 用户启用一个依赖尚未满足的模块
- **THEN** 系统保存期望启用状态但不加载入口，并显示等待依赖诊断

#### Scenario: 停用存在依赖者的模块
- **WHEN** 用户尝试停用一个仍被期望启用模块必需依赖的模块
- **THEN** 系统拒绝操作并返回受影响依赖者，现有激活计划保持不变

#### Scenario: 显式安全回滚
- **WHEN** 用户回滚一个存在上一计划版本的模块，且该版本与其他当前选择兼容并不要求隐式改变其他模块
- **THEN** 系统提交新的计划代次，并在重新加载后使用回滚版本

#### Scenario: 回滚会破坏依赖者
- **WHEN** 目标旧版本不满足当前依赖者或需要同时改变其他模块
- **THEN** 系统拒绝单模块回滚、列出受影响模块，并保持当前计划

#### Scenario: 卸载存在依赖者的模块
- **WHEN** 用户确认卸载一个仍被期望启用或等待模块必需依赖的模块
- **THEN** 系统拒绝卸载、返回依赖者列表，并保留该模块的代码、计划状态和数据库

#### Scenario: 卸载无依赖者模块
- **WHEN** 用户确认卸载一个没有必需依赖者的运行时模块
- **THEN** 系统从计划和磁盘删除该模块所有代码版本及安装状态，在重新加载后移除其贡献，并保留模块数据库供重新安装或显式清理

#### Scenario: 显示已卸载模块保留数据
- **WHEN** 模块已经卸载但其数据库仍存在
- **THEN** 模块管理界面显示该模块 ID、数据库占用和需要明确确认的清理操作

#### Scenario: 操作失败可见
- **WHEN** 安装、启用、停用、升级、回滚、卸载或数据清理操作失败或等待依赖
- **THEN** 模块管理界面显示结构化原因，且最后一个完整可用计划和未被明确清理的数据保持可用
