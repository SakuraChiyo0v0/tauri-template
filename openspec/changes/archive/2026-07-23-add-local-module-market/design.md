## Context

运行时模块只能通过 Host SDK 访问受控能力。当前基座可以由内置模块管理页安装用户选择的单个 `.mtp`，但普通模块既不能请求目录授权，也不能检查或安装目录中的包。把市场 UI 写进基座会违背可拆卸模块原则；把原始路径和 `install(packagePath)` 暴露给模块则会允许静默安装任意本地代码。

## Goals / Non-Goals

**Goals:**

- 市场 UI、目录记忆、筛选和交互全部属于独立 `.mtp` 模块。
- 基座只提供目录选择、受限扫描、包校验与安装的公共安全边界。
- 用户可更换市场目录，模块只能看到不透明 grant ID 和目录显示名。
- 扫描与安装复用现有包校验、依赖解析、权限审批和回滚行为。
- V2–V4 模块继续运行；新模板和市场模块使用 SDK V5。

**Non-Goals:**

- 不支持网络源、后台监控、自动安装、批量依赖下载、包上传或发布。
- 不在首版增加发布者签名或信任链。
- 不让市场模块读取包内任意文件、真实路径或基座内部 store。

## Decisions

### 1. 新增 SDK V5 `moduleRepository`，而不是公开内部安装 API

V5 提供 `chooseDirectory()`、`scan(grantId)` 和 `install(grantId, fileName)`。目录选择由基座调用系统选择器；Rust 只接受会话令牌、不透明 grant ID 和单层文件名。模块无法构造路径或访问仓库外文件。

扫描返回经过同一 Rust 清单解析器验证的安全 DTO，包括包文件名、双语名称与说明、版本、依赖、权限摘要、当前安装版本和可观察状态。它不返回入口源码，也不执行包内容。

### 2. 扫描和安装属于一项显式高权限能力

清单新增 `nativeCapabilities.moduleRepository: { install: true }`。该声明进入权限指纹和用户可读摘要；没有批准的模块不能获得会话，旧批准也不会自动覆盖新增权限。

曾考虑只依赖外部目录读取权限，但读取仓库与安装可执行模块的风险不同，必须单独展示。

### 3. 目录授权复用现有 grant 存储

目录选择产生模块专属、只读且可列举的目录 grant。市场模块把 grant ID 存进自己的设置；路径只保存在 Rust grant 文件中。撤销后扫描和安装立即失败，重新选择会创建新 grant。

### 4. 安装仍由 `ModuleStore` 完成

仓库安装命令在每次操作时重新解析授权和包文件名，再调用 `install_with_plan`。因此版本不可变、依赖解析、权限等待、失败恢复与旧版安装入口保持一致。包在扫描后被替换也不能绕过最终安装校验。

### 5. 首版按包逐个安装

市场展示依赖，但一次只提交一个包。依赖未安装时模块可以先进入等待状态，用户再安装依赖；这与当前解析器行为一致，避免首版引入跨包事务和隐式级联安装。

## Risks / Trade-offs

- [获批市场模块可以安装授权目录中的任意合法包] → 权限摘要明确标为模块安装能力，目录由用户选择，且每次安装仍执行完整校验。
- [扫描结果与点击安装之间文件可能变化] → 安装时重新读取、校验并计算包哈希，不信任扫描缓存。
- [同一模块多个版本会占据多张卡片] → 市场模块按模块 ID 分组并优先展示最高版本，首版仍允许查看其他版本。
- [Host SDK 升级影响模板] → 基座保留 V2–V4 分支，只有模板默认版本和新市场模块切换到 V5。

## Migration Plan

1. 基座先发布 SDK V5 和清单解析支持，V2–V4 行为保持不变。
2. 独立模板同步 V5 类型和模拟 API。
3. 打包并通过现有模块管理页首次安装 `local-module-market`，批准目录与安装权限。
4. 用户在市场模块中选择本地仓库目录；撤销权限或卸载市场模块即可回滚，不影响已安装模块。

## Open Questions

无。远程源、签名和批量依赖安装留待后续独立变更。
