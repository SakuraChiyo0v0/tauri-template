# Design: module-event-bus (Host SDK V7)

## 目标

- 让运行时模块可以在数据变化后主动通知其他模块，消除轮询和手动刷新。
- 复用既有模块服务的信任边界、数据边界和生命周期管理，不引入第二种风格。
- V2–V6 模块行为完全不变。

## 非目标

- 不做事件持久化、离线回放或投递保证；模块未激活期间的事件直接丢弃。
- 不做通配符订阅、事件过滤表达式或优先级。
- 不做跨窗口/跨进程事件；事件只存在于当前 WebView 内。
- 不改变模块服务的请求-响应语义；事件没有返回值。
- 不在模块管理页新增事件可视化（清单声明已提供静态透明度，可视化为后续可选项）。

## 关键决策

### 1. 订阅不要求声明模块依赖

服务的调用方必须把提供者声明为依赖，因为调用有明确目标。事件的价值恰好在解耦：发布者不知道也不应知道订阅者是谁。如果要求订阅者声明依赖，会把安装顺序和依赖求解耦合进订阅关系，通用型订阅者（例如未来的日志聚合模块）也无法声明尚未存在的发布者。

决策：订阅按事件 ID 全局匹配，但模块必须在清单 `events.subscribes` 中声明事件 ID，发布必须在 `events.publishes` 中声明。声明提供静态透明度，匹配提供运行时解耦。信任边界与模块服务一致：所有模块共享同一 WebView，本边界针对“可信但可能有缺陷”的模块，不是恶意代码沙箱。

### 2. 事件 ID 复用服务 ID 格式

事件 ID 使用与服务 ID 相同的格式（如 `notes.changed.v1`），约定 `<域>.<动作>.v<主版本>`。理由：一套命名规则降低学习成本；带主版本使不兼容的事件载荷变更可以换新 ID，与版本化服务的演进方式一致。

### 3. 载荷复用服务的数据边界

事件载荷与发布信封（envelope）使用和 `services.call()` 完全相同的受限 JSON 值校验、深复制、深度（32）与大小（256KB）限制。理由：避免在 Host SDK 中出现第二种数据边界；模块作者只需理解一套规则。

发布信封由底座构造：`{ eventId, publisherModuleId, publishedAt, payload }`。`publisherModuleId` 由底座从会话身份注入，模块无法伪造来源。

### 4. 投递语义：异步、尽力而为、异常隔离

`publish()` 在底座校验（已声明、ID 合法、载荷合法）后立即返回；投递异步执行。单个订阅者监听器抛出异常不影响其他订阅者，异常记录到底座日志（带模块与事件上下文，不记录载荷内容）。同一发布者的事件按发布顺序投递（FIFO），不保证跨发布者的全局顺序。

理由：事件是通知而非事务；把订阅者故障与发布者隔离，避免一个有缺陷的模块阻塞整个总线。

### 5. SDK 形态与生命周期

V7 模块获得 `sdk.events`：

- `publish(eventId, payload?)` — 校验通过后入队投递。
- `subscribe(eventId, listener)` — 返回退订函数；重复订阅同一事件允许，各自独立退订。

事件总线实现为前端核心模块（与服务总线并列），订阅注册按模块会话跟踪；`releaseRuntimeModuleHostSdk` 时自动退订该模块全部订阅，停用、卸载、版本切换与计划重载走同一路径。发布是无状态的（按清单即时校验），不需要显式注册/注销主题。

### 6. Rust 侧只做清单校验

事件总线不需要新的原生命令。Rust 侧改动仅为：清单解析接受 `sdkVersion: 7` 与 `events` 字段（V7 以下声明事件报错）、`MAX_SDK_VERSION` 提升到 7。

### 7. 附带 bug 修复（已确认根因）

`src-tauri/src/features/runtime_modules/mod.rs` 的 `require_active_database_module` 使用 `2..=5` 检查 SDK 版本，V6 模块调用数据库命令会被错误拒绝。根因：V6 升级时该范围未同步。修复：改为覆盖 MIN–MAX 支持范围，并补 V6（及 V7）模块数据库访问的回归测试。这属于恢复 V6 规格既定行为，不是新行为设计。

## 考虑过的替代方案

- **复用服务总线模拟事件**（消费者轮询 `available()` + 定时 `call()`）：不改变底座，但正是本次要消除的过期数据与浪费调用问题，否决。
- **订阅必须声明依赖**：更严格的可见性，但破坏解耦价值并耦合安装顺序，见决策 1，否决。
- **持久化事件流**：能支持回放与审计，但引入存储、保留策略和隐私问题，远超当前需求，列为非目标。

## 风险与权衡

- 任意模块可订阅任意已声明事件：事件载荷应对宿主内所有模块可见，发布者不得放入机密数据；这与既有“非恶意沙箱”信任边界一致，将在规格与模板文档中明示。
- 事件风暴（高频发布拖慢 WebView）：依靠 256KB 载荷限制与模块作者自律；不引入速率限制，保持实现简单，后续如有真实案例再评估。
