## Why

运行时模块目前只有“请求-响应”式的模块服务（`services.call()`）：消费者必须主动调用提供者才能拿到最新数据。以便签面板为例，用户修改便签后，面板不知道数据已变化，只能手动刷新或轮询。模块之间缺少一种“数据变化后主动通知”的机制，导致协作模块的页面容易显示过期内容。

## What Changes

- 新增模块事件总线：模块可以发布“某件事已经发生”的通知（事件），其他模块可以按事件 ID 订阅并自动收到通知。当前情况：便签面板只能手动刷新 → 改造后：便签模块保存数据后发布事件，面板收到事件立即自动刷新。
- 事件是单向通知，不返回结果；需要返回值的场景仍使用既有模块服务，两者互不替代。
- 模块必须在清单中显式声明自己会发布哪些事件、想订阅哪些事件；未声明的发布会被拒绝，未声明的订阅也会被拒绝。用户和审查者可以通过清单看清模块的事件行为。
- 事件携带的数据沿用模块服务相同的受限 JSON 边界（拒绝函数、循环引用、超限数据），投递前由底座深复制，模块之间不共享引用。
- 发布者身份（哪个模块发的、什么时间发的）由底座附加，模块不能伪造来源。
- 订阅按事件 ID 全局匹配，**不要求**订阅者把发布者声明为模块依赖；事件在模块未激活期间直接丢弃，不做持久化和回放。
- Host SDK 从 V6 升级到 V7：V7 = V6 全部能力 + `events` 命名空间；V2–V6 模块行为完全不变。
- 同步升级独立模块模板（SDK 类型、双语清单示例、模拟宿主事件支持），并用真实参考模块验证：`local-notes` 在便签增删改后发布事件，`notes-dashboard` 订阅事件自动刷新统计（保留手动刷新）。
- 附带修复一个已确认根因的 bug：Rust 侧数据库命令的版本检查停留在 `2..=5`，导致 SDK V6 模块调用数据库被错误拒绝；本次把范围修正为覆盖 V6（及 V7）并补回归测试。

## Capabilities

### New Capabilities

- `runtime-module-events`: SDK V7 模块事件的清单声明、发布授权、订阅匹配、数据隔离、投递语义与生命周期清理。

### Modified Capabilities

- `standalone-runtime-module-development`: 模板 Host SDK 类型与模拟宿主从 V5 描述更新到 V7，新增事件总线预览能力（同时补记 V6 期间未同步的规格漂移）。
- `reference-module-suite`: `local-notes` 在便签变更后发布事件；`notes-dashboard` 订阅事件自动刷新统计，保留手动刷新与既有服务调用。

## Impact

- 基座前端：`src/core/runtime-modules/` 新增事件总线模块与测试，清单解析、SDK 构建与释放路径扩展 V7 分支。
- 基座原生：`src-tauri` 清单校验接受 SDK V7 与 `events` 声明，修正数据库命令版本范围检查。
- 独立仓库：`tauri-module-template`（SDK 类型、清单示例、模拟宿主、测试）、`tauri-modules/local-notes` 与 `tauri-modules/notes-dashboard`（事件发布/订阅与版本升级），生成的新 `.mtp` 复制到 `tauri-module-market`。
- 不影响：源码模块注册表、模块包生命周期、权限审批、依赖求解与本地模块仓库行为；不引入远程下载、动态 Rust 或新的原生权限。
