# module-package-lifecycle Specification

## Purpose

定义本地运行时模块包从校验、安装、升级到失败恢复、回滚和卸载的可观察行为与安全边界，并确保损坏、不兼容或恶意构造的包不会破坏现有可用版本及模块目录之外的数据。
## Requirements
### Requirement: 本地模块包必须经过完整校验
系统 MUST 只安装用户主动选择且符合受支持包协议的 `.mtp` 文件。安装服务 MUST 在修改活动版本前校验 ZIP 结构、清单字段、模块标识、语义版本、宿主版本范围、SDK 版本、原生能力声明、入口文件、路径安全、文件数量和展开大小。V3 原生能力声明无效或包含底座禁止范围时 MUST 在保存版本前拒绝。

#### Scenario: 安装有效的新模块
- **WHEN** 用户选择一个结构有效、版本兼容且模块 ID 尚未安装的无原生权限模块
- **THEN** 系统将其内容写入该模块版本目录，并按依赖与权限状态决定是否选择该版本

#### Scenario: 安装有效 V3 权限模块
- **WHEN** 用户选择结构有效且原生能力声明合法的 V3 模块
- **THEN** 系统保存模块版本和规范化权限摘要，但在用户批准前不激活该版本

#### Scenario: 拒绝不兼容底座的模块
- **WHEN** 模块清单声明的 `hostVersion` 不包含当前底座版本
- **THEN** 系统拒绝安装并显示兼容性错误，且不创建或切换活动版本

#### Scenario: 拒绝越界路径
- **WHEN** 包中包含绝对路径、父目录跳转、符号链接或解压后超出模块目录的条目
- **THEN** 系统拒绝整个包，且模块安装目录之外的文件 MUST 保持不变

#### Scenario: 拒绝危险原生能力声明
- **WHEN** V3 清单请求任意 Shell、提权、HKLM 写入、未知能力或无边界系统范围
- **THEN** 系统拒绝整个包，且既有版本、权限批准和激活计划保持不变

#### Scenario: 校验失败不影响旧版本
- **WHEN** 已安装模块的新包缺少入口、清单无效、权限声明无效或完整性检查失败
- **THEN** 原活动版本继续保持活动，且失败包不得出现在可用版本列表中

### Requirement: 模块升级必须保留可回滚版本
系统 SHALL 在安装更高且兼容的新版本时保留当前活动版本，并先把新版本完整写入不可变版本目录。系统 MUST 只在整张期望启用模块依赖图存在兼容解时原子提交新的全局激活计划；无解的新版本保留为等待选择的可用版本，当前计划保持不变。

#### Scenario: 升级已安装模块
- **WHEN** 用户安装同一模块 ID 的更高版本，并且新版本可与所有期望启用模块形成兼容组合
- **THEN** 系统原子提交新选择组合，原活动版本保留为可用版本，且上一整套选择计划可用于安全回滚判断

#### Scenario: 协调升级多个等待版本
- **WHEN** 新安装版本使多个先前等待的版本共同形成兼容依赖组合
- **THEN** 系统在一个计划代次中同时切换这些模块，不暴露部分切换状态

#### Scenario: 新版本依赖无解
- **WHEN** 新版本无法与当前已安装版本形成完整兼容组合
- **THEN** 包安装成功但新版本保持等待，当前活动组合不变，管理界面显示依赖诊断

#### Scenario: 拒绝重复或较低版本覆盖
- **WHEN** 用户提交已经存在于该模块版本目录中的版本
- **THEN** 系统拒绝覆盖相同不可变版本；较低但尚未保存的版本只可作为可用版本保存，不得自动替换当前选择

### Requirement: 激活失败必须安全恢复
系统 MUST 接收前端激活结果。新版本激活失败时，有上一版本则恢复上一版本；没有上一版本则阻止失败版本再次自动激活，并保留可诊断错误。

#### Scenario: 升级版本激活失败
- **WHEN** 新活动版本的入口导入、`activate` 调用或页面元素校验失败，并且存在上一版本
- **THEN** 系统恢复上一版本为活动版本、记录失败版本和错误，并允许底座重新加载恢复后的模块

#### Scenario: 首次安装后激活失败
- **WHEN** 首次安装的模块激活失败且不存在上一版本
- **THEN** 系统保留安装记录但将该版本标记为激活受阻，底座继续运行且不反复加载该版本

### Requirement: 用户必须能够管理模块生命周期
模块管理界面 SHALL 展示每个运行时模块的选择版本、可用版本、期望启用状态、依赖、依赖者、最近错误、数据库占用、原生权限状态和运行资源，并提供与当前依赖图和权限状态相符的批准、撤销、启用、停用、回滚和卸载操作。系统 MUST 在提交操作前分析直接与间接影响，且不得静默批准新权限、级联修改其他模块或遗留模块系统资源。普通卸载 MUST 删除模块代码和安装状态但默认保留模块数据库、权限批准和外部 grant；已卸载模块的保留数据 MUST 继续可见并可由用户显式清理。

#### Scenario: 批准等待权限模块
- **WHEN** 用户查看完整权限摘要并批准等待中的 V3 模块
- **THEN** 系统保存批准记录并重新解析激活计划，只有依赖也满足时才激活模块

#### Scenario: 启用等待中的模块
- **WHEN** 用户启用一个依赖或权限尚未满足的模块
- **THEN** 系统保存期望启用状态但不加载入口，并显示等待依赖或等待权限诊断

#### Scenario: 停用存在依赖者的模块
- **WHEN** 用户尝试停用一个仍被期望启用模块必需依赖的模块
- **THEN** 系统拒绝操作并返回受影响依赖者，现有激活计划和运行资源保持不变

#### Scenario: 显式安全回滚
- **WHEN** 用户回滚到权限已批准、存在上一计划版本且与当前依赖图兼容的版本
- **THEN** 系统释放当前版本运行资源、提交新计划代次并在重新加载后使用回滚版本

#### Scenario: 回滚版本权限未批准
- **WHEN** 目标回滚版本声明的权限没有有效批准记录
- **THEN** 系统拒绝激活该版本并显示需要审查的权限，不复用当前版本会话或权限

#### Scenario: 回滚会破坏依赖者
- **WHEN** 目标旧版本不满足当前依赖者或需要同时改变其他模块
- **THEN** 系统拒绝单模块回滚、列出受影响模块，并保持当前计划

#### Scenario: 撤销活动模块权限
- **WHEN** 用户确认撤销活动模块的原生权限
- **THEN** 系统先安全停用模块并释放会话、托盘、快捷键和进行中操作，再保存撤销状态

#### Scenario: 卸载存在依赖者的模块
- **WHEN** 用户确认卸载一个仍被期望启用或等待模块必需依赖的模块
- **THEN** 系统拒绝卸载、返回依赖者列表，并保留该模块代码、计划、权限和运行状态

#### Scenario: 卸载无依赖者模块
- **WHEN** 用户确认卸载一个没有必需依赖者的运行时模块
- **THEN** 系统释放全部运行资源、从计划和磁盘删除模块代码，并默认保留模块数据库、权限批准和外部 grant

#### Scenario: 显示已卸载模块保留数据
- **WHEN** 模块已经卸载但其数据库仍存在
- **THEN** 模块管理界面显示该模块 ID、数据库占用和需要明确确认的清理操作

#### Scenario: 操作失败可见
- **WHEN** 安装、权限批准、撤销、启用、停用、升级、回滚或卸载操作失败或等待条件
- **THEN** 模块管理界面显示结构化原因，最后一个完整可用计划及未被明确撤销的授权保持可用

### Requirement: 安装状态必须可恢复
模块版本目录与模块目录元数据 MUST 先于全局激活计划完整写入。全局计划 MUST 使用防止部分写入的方式保存，并以单调增加的代次标识完整选择组合。进程在包复制、迁移或计划切换期间中断时，下一次启动 SHALL 恢复最后一个完整计划并把额外完整版本视为未选择版本。

#### Scenario: 安装过程中进程中断
- **WHEN** 应用在暂存包内容完成前退出
- **THEN** 下一次启动继续使用原全局计划，并安全清理未完成暂存目录

#### Scenario: 版本落盘后计划写入前中断
- **WHEN** 新版本目录已经完整保存但新激活计划尚未提交时应用退出
- **THEN** 下一次启动继续使用旧计划，并把新版本列为可用但未选择版本

#### Scenario: 计划写入过程中断
- **WHEN** 应用在替换全局激活计划时意外退出
- **THEN** 下一次启动读取旧计划或完整新计划之一，不得读取部分选择组合

#### Scenario: 迁移旧逐模块状态
- **WHEN** 存在 V1 模块 `state.json` 但没有全局激活计划
- **THEN** 系统优先沿用旧活动版本建立初始计划，吸收前端提供的旧禁用模块，并保证重复迁移幂等
