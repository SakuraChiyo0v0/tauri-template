# runtime-module-contributions Specification

## Purpose

定义运行时模块被发现、激活并向共享侧边栏、设置、主题和日志系统提供贡献时必须遵守的宿主协议，使独立模块无需修改应用外壳即可安全接入，并在启停、失败和版本切换时保持底座可用。

## Requirements

### Requirement: 底座必须发现运行时模块
底座启动时 MUST 从 Rust 安装服务读取所有活动运行时模块的已验证清单，并把它们注册到可观察模块聚合层。运行时模块不得要求修改应用外壳或静态模块注册文件。

#### Scenario: 启动时发现活动模块
- **WHEN** 应用数据目录中存在状态完整且兼容的活动模块
- **THEN** 模块管理器显示该模块，并根据其启用状态聚合页面和设置贡献

#### Scenario: 无运行时模块时启动
- **WHEN** 应用数据目录中没有安装运行时模块
- **THEN** 内置系统模块和日志模块继续正常初始化，且底座不显示虚构的演示模块

### Requirement: 运行时贡献必须由清单驱动
运行时模块 SHALL 通过清单声明导航和基础设置贡献。底座 MUST 使用共享外壳渲染导航、页面标题、设置控件和主题令牌，不允许模块直接修改共享设置页面或侧边栏代码。

#### Scenario: 注册导航页面
- **WHEN** 已启用模块清单包含合法且不冲突的导航贡献
- **THEN** 对应入口出现在清单指定的导航分组和顺序，并打开通用运行时模块页面容器

#### Scenario: 注册基础设置
- **WHEN** 已启用模块清单包含合法的 `switch` 或 `select` 设置
- **THEN** 设置页面按模块 ID 命名空间显示并持久化这些设置

#### Scenario: 贡献标识冲突
- **WHEN** 运行时模块的模块 ID、导航 ID 或自定义元素标签与已注册贡献冲突
- **THEN** 系统拒绝激活该运行时模块，显示冲突错误，并保持其他模块可用

### Requirement: 模块代码必须通过稳定入口激活
每个运行时模块 MUST 提供清单声明的单文件 ESM 入口，并导出 `activate(hostSdk)`。成功激活后，入口 MUST 定义每个导航贡献声明的自定义元素。

#### Scenario: 成功激活模块入口
- **WHEN** 已启用模块的 ESM 入口成功导入、`activate` 成功返回且所有声明元素已定义
- **THEN** 模块状态变为已就绪，其页面可在共享内容区域中渲染

#### Scenario: 入口缺少激活函数
- **WHEN** 模块入口没有导出可调用的 `activate`
- **THEN** 激活失败并进入模块包生命周期规定的失败恢复流程

#### Scenario: 页面元素缺失
- **WHEN** `activate` 返回后仍有清单声明的自定义元素未定义
- **THEN** 激活失败，缺失元素名称出现在诊断错误中

### Requirement: 宿主 SDK 必须提供最小稳定能力
底座 SHALL 向 `activate` 传入带版本的宿主 SDK。V1 SDK MUST 提供模块身份、底座版本、分级日志、本模块设置读写和主题状态订阅，且不得暴露内部注册表或任意文件系统写入接口。

#### Scenario: 模块记录日志
- **WHEN** 运行时模块通过宿主 SDK 写入分级日志
- **THEN** 日志使用该模块 ID 作为来源进入统一日志管道和日志控制台

#### Scenario: 模块读写自身设置
- **WHEN** 模块通过 SDK 读写设置
- **THEN** 设置键被限制在该模块 ID 命名空间，并触发现有设置订阅更新

#### Scenario: 主题发生变化
- **WHEN** 用户切换底座主题模式或主题预设
- **THEN** 运行时页面继承最新 CSS 主题变量，已订阅模块收到主题状态变化通知

### Requirement: 启停必须同步改变贡献和生命周期
用户停用运行时模块时，系统 MUST 从导航和设置聚合结果中移除其贡献并调用可选停用生命周期；重新启用时 MUST 重新激活并恢复贡献。

#### Scenario: 停用运行时模块
- **WHEN** 用户在模块管理器关闭一个可停用运行时模块
- **THEN** 其导航和设置立即消失，当前位于该模块页面时回退到第一个可用页面

#### Scenario: 重新启用运行时模块
- **WHEN** 用户重新启用已停用的运行时模块
- **THEN** 系统执行激活检查，并在成功后恢复其导航和设置贡献

### Requirement: 版本切换必须清理旧运行上下文
安装升级、回滚或卸载成功后，底座 SHALL 重新加载 WebView，再从磁盘活动状态重新发现模块，避免在同一 JavaScript 上下文混用两个模块版本。

#### Scenario: 升级后重新加载
- **WHEN** 新模块版本成功安装并设为活动版本
- **THEN** 底座重新加载，之后只导入新活动版本的入口

#### Scenario: 回滚后重新加载
- **WHEN** 用户回滚或系统因激活失败恢复上一版本
- **THEN** 底座重新加载，之后只导入恢复版本的入口
