# runtime-module-contributions Specification

## Purpose

定义运行时模块被发现、激活并向共享侧边栏、设置、主题和日志系统提供贡献时必须遵守的宿主协议，使独立模块无需修改应用外壳即可安全接入，并在启停、失败和版本切换时保持底座可用。

## Requirements

### Requirement: 底座必须发现运行时模块
底座启动时 MUST 从 Rust 安装服务读取所有已安装运行时模块以及已解析的全局激活计划。底座 SHALL 只注册计划选择且未被依赖阻止的模块贡献，并使用计划状态展示等待或失败模块；运行时模块不得要求修改应用外壳或静态模块注册文件。

#### Scenario: 启动时发现完整依赖图
- **WHEN** 应用数据目录中存在状态完整且依赖可解的活动模块
- **THEN** 模块管理器显示所有已安装模块，并按计划拓扑顺序激活选择模块和聚合贡献

#### Scenario: 发现等待依赖模块
- **WHEN** 已安装模块期望启用但未被全局计划选择
- **THEN** 模块管理器显示该模块及结构化依赖诊断，侧边栏和设置不显示其贡献

#### Scenario: 无运行时模块时启动
- **WHEN** 应用数据目录中没有安装运行时模块
- **THEN** 内置系统模块和日志模块继续正常初始化，且底座不显示虚构的演示模块

#### Scenario: 迁移旧前端启用状态
- **WHEN** 首次创建全局计划且旧前端状态中存在已停用运行时模块
- **THEN** 启动引导在激活入口前把这些模块作为禁用迁移给 Rust，之后以 Rust 计划为唯一运行时启用状态

### Requirement: 运行时贡献必须由清单驱动
运行时模块 SHALL 通过清单声明导航和基础设置贡献。底座 MUST 使用共享外壳渲染导航、页面标题、设置控件和主题令牌，不允许模块直接修改共享设置页面或侧边栏代码。

#### Scenario: 注册导航页面
- **WHEN** 已启用模块清单包含合法且不冲突的导航贡献
- **THEN** 对应入口出现在清单指定的导航分组和顺序，并打开通用运行时模块页面容器

#### Scenario: 注册基础设置
- **WHEN** 已启用模块清单包含合法的 `switch` 或 `select` 设置
- **THEN** 设置页面按模块 ID 命名空间显示并持久化这些设置

#### Scenario: 贡献标识冲突
- **WHEN** 运行时模块的模块 ID、导航 ID 或自定义元素标签与已注册贡献冲突
- **THEN** 系统拒绝激活该运行时模块，显示冲突错误，并保持其他模块可用

### Requirement: 模块代码必须通过稳定入口激活
每个运行时模块 MUST 提供清单声明的单文件 ESM 入口，并导出 `activate(hostSdk)`。成功激活后，入口 MUST 定义每个导航贡献声明的自定义元素。

#### Scenario: 成功激活模块入口
- **WHEN** 已启用模块的 ESM 入口成功导入、`activate` 成功返回且所有声明元素已定义
- **THEN** 模块状态变为已就绪，其页面可在共享内容区域中渲染

#### Scenario: 入口缺少激活函数
- **WHEN** 模块入口没有导出可调用的 `activate`
- **THEN** 激活失败并进入模块包生命周期规定的失败恢复流程

#### Scenario: 页面元素缺失
- **WHEN** `activate` 返回后仍有清单声明的自定义元素未定义
- **THEN** 激活失败，缺失元素名称出现在诊断错误中

### Requirement: 宿主 SDK 必须提供最小稳定能力
底座 SHALL 根据模块清单声明的 SDK 版本向 `activate` 传入对应的带版本宿主 SDK。V1 SDK MUST 保持模块身份、底座版本、分级日志、本模块设置读写和主题状态订阅，且不得出现 V2 专属数据库成员。V2 SDK MUST 保留 V1 能力并增加本模块隔离数据库接口。任何版本均不得暴露内部注册表、任意文件系统路径或原始 Tauri invoke。

#### Scenario: V1 模块保持原 SDK
- **WHEN** 已安装模块清单声明 `sdkVersion: 1`
- **THEN** 底座传入字面量版本为 1 且不包含数据库成员的 V1 SDK，现有模块无需修改即可激活

#### Scenario: V2 模块获得数据库能力
- **WHEN** 已安装模块清单声明 `sdkVersion: 2`
- **THEN** 底座传入字面量版本为 2 且包含模块私有数据库接口的 V2 SDK

#### Scenario: 拒绝不支持的 SDK 版本
- **WHEN** 模块清单声明底座不支持的 SDK 版本
- **THEN** 安装校验失败，现有模块版本和激活计划保持不变

#### Scenario: 模块记录日志
- **WHEN** 运行时模块通过宿主 SDK 写入分级日志
- **THEN** 日志使用该模块 ID 作为来源进入统一日志管道和日志控制台

#### Scenario: 模块读写自身设置
- **WHEN** 模块通过 SDK 读写设置
- **THEN** 设置键被限制在该模块 ID 命名空间，并触发现有设置订阅更新

#### Scenario: 主题发生变化
- **WHEN** 用户切换底座主题模式或主题预设
- **THEN** 运行时页面继承最新 CSS 主题变量，已订阅模块收到主题状态变化通知

### Requirement: 启停必须同步改变贡献和生命周期
用户启用或停用运行时模块时，系统 MUST 通过 Rust 全局计划改变其期望启用状态并重新解析依赖图。停用不得破坏仍期望启用的必需依赖者；重新启用只有在进入选择计划并成功激活后才能恢复贡献。

#### Scenario: 停用无依赖者运行时模块
- **WHEN** 用户在模块管理器关闭一个不被其他期望启用模块必需依赖的模块
- **THEN** 新计划移除该模块选择，其导航和设置消失，并调用可选停用生命周期

#### Scenario: 停用必需 provider
- **WHEN** 用户关闭一个仍有期望启用依赖者的模块
- **THEN** 系统拒绝停用、显示依赖者列表，现有页面和设置保持不变

#### Scenario: 重新启用且依赖满足
- **WHEN** 用户重新启用模块且解析器找到完整兼容组合
- **THEN** 系统提交新计划、按拓扑顺序激活，并在成功后恢复其导航和设置贡献

#### Scenario: 重新启用但依赖缺失
- **WHEN** 用户重新启用模块但必需依赖仍不满足
- **THEN** 系统保存期望启用状态并显示等待依赖，不执行入口或恢复贡献

### Requirement: 版本切换必须清理旧运行上下文
安装、依赖解析、启停、回滚、激活失败恢复或卸载导致全局计划选择版本改变后，底座 SHALL 重新加载 WebView，并只按新计划代次和拓扑顺序导入入口，避免在同一 JavaScript 上下文混用版本或保留失效依赖引用。

#### Scenario: 协调升级后重新加载
- **WHEN** 一个新计划同时切换一个或多个模块选择版本
- **THEN** 底座只重新加载一次，之后仅导入新计划选择的版本

#### Scenario: 启停改变计划后重新加载
- **WHEN** 启用或停用操作成功改变选择模块集合
- **THEN** 底座重新加载并按新拓扑顺序恢复贡献

#### Scenario: 激活失败重新求解
- **WHEN** 模块激活失败使 Rust 阻止该版本并生成不同的兼容计划
- **THEN** 底座只自动刷新一次，随后按新计划加载；依赖失败模块不会在同一代次反复执行

#### Scenario: 无选择变化时不刷新
- **WHEN** 包只作为等待版本保存，或操作被依赖影响检查拒绝
- **THEN** 当前 WebView 与活动贡献保持不变，管理页面就地更新诊断
